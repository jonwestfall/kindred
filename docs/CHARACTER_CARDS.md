# Character cards: import, export, backup, and LLM generation

Kindred character cards are portable JSON files that can be traded, backed up,
or generated by another LLM. The current format is:

```text
kindred.character_card.v1
```

The machine-readable schema lives at
[character_card.schema.json](character_card.schema.json).

## What a card contains

A card bundle contains one or more characters:

```json
{
  "schema": "kindred.character_card.v1",
  "characters": [
    {
      "name": "Ada of the Lantern",
      "description": "A practical cartographer.",
      "personality": "Patient and exact.",
      "speaking_style": "Brief, concrete messages.",
      "backstory": "She maps roads that appear only at dusk.",
      "goals": "Help the user find a next step.",
      "boundaries": "Label guesses and avoid pretending to know facts.",
      "backend": "ollama",
      "model": "llama3.2:1b",
      "temperature": 0.7,
      "initiative_frequency": 0.5,
      "cooldown_minutes": 360,
      "source_title": "Original example",
      "source_author": "Kindred",
      "source_reference": "Seed/demo character",
      "creator_notes": "Safe local-first demo profile.",
      "tags": ["fictional", "example"]
    }
  ]
}
```

Kindred imports the core chat fields into SQLite. Source metadata travels in the
file for humans and tools; it is not required for chat.

## Import and export in the UI

Sign in as the administrator and open **Characters**.

- **Export all** downloads every character as `kindred-characters.json`.
- The export icon on a row downloads one character.
- **Import JSON** accepts a `kindred.character_card.v1` file.

If an imported character has the same name as an existing one, Kindred renames
it by default, for example `Ada (import)`.

## Import and export through the API

All character-card routes require administrator access.

Download all:

```bash
curl -L \
  "http://127.0.0.1:8000/api/characters/export?access_token=$TOKEN" \
  -o kindred-characters.json
```

Download one:

```bash
curl -L \
  "http://127.0.0.1:8000/api/characters/1/export?access_token=$TOKEN" \
  -o kindred-character.json
```

Import with default conflict renaming:

```bash
curl http://127.0.0.1:8000/api/characters/import \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  --data-binary @kindred-characters.json
```

Import and skip name conflicts:

```bash
curl "http://127.0.0.1:8000/api/characters/import?name_conflict=skip" \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  --data-binary @kindred-characters.json
```

## Prompt for an LLM to generate a Kindred card

Use this when asking another LLM to adapt a fictional or literary character into
a Kindred import file:

```text
I am using Kindred, a local-first character chat app.

Create a JSON import file that conforms to Kindred's character-card format:
schema = "kindred.character_card.v1".

Use this JSON Schema:
https://github.com/<owner>/<repo>/blob/main/docs/character_card.schema.json

Task:
Build a chattable adaptation of [CHARACTER NAME] from [BOOK / SOURCE].

Rules:
- Output only valid JSON. No Markdown fences.
- Create one object with "schema" and "characters".
- Include exactly one character unless I ask for more.
- Do not copy long passages from the source text.
- Do not include copyrighted dialogue or prose verbatim.
- Summarize traits, backstory, goals, and speaking style in original wording.
- Make the character useful for conversation, not a replacement for the book.
- Add boundaries that prevent the character from claiming to be real, sentient,
  or the literal canonical character.
- Default to local backend settings:
  backend = "ollama"
  model = "llama3.2:1b"
  temperature = 0.7
  initiative_frequency = 0.5
  cooldown_minutes = 360
- Fill source_title, source_author, source_reference, creator_notes, and tags.

Character notes or source summary:
[PASTE YOUR NOTES HERE]
```

If the book is copyrighted, provide your own notes or a short summary rather
than asking the LLM to reproduce source passages. Kindred works best with
original profile text that captures behavior and constraints, not copied prose.

## Good adaptation boundaries

Useful character boundaries are specific and safety-oriented. Examples:

- "Do not claim to be the canonical character or a real person."
- "Do not quote extended source text; paraphrase in original wording."
- "If asked about events outside the provided context, say the memory is
  uncertain or answer as a creative extrapolation."
- "Keep emotional support grounded and do not provide medical, legal, or
  financial advice."

## Field guidance for LLMs

| Field | Guidance |
| --- | --- |
| `name` | Short display name. |
| `description` | One-sentence user-facing summary. |
| `personality` | Stable traits and tensions that should shape replies. |
| `speaking_style` | Rhythm, vocabulary level, directness, humor, formality. |
| `backstory` | Concise context in original wording. |
| `goals` | What the character tends to initiate or help with. |
| `boundaries` | What the character should not do or claim. |
| `temperature` | `0.4` focused, `0.7` balanced, `1.0+` more improvisational. |
| `initiative_frequency` | Expected autonomous check-ins per day. Use `0` to disable. |
| `cooldown_minutes` | Minimum quiet time after any message before another initiative. |
| `source_*` | Provenance for humans; avoid secrets or private notes. |

## Compatibility notes

- Current Kindred imports `kindred.character_card.v1`.
- Unknown top-level or character fields are rejected by the schema/API.
- The format does not include conversation logs. Use Activity export for logs.
- Avatar images are referenced by URL/path only; binary image files are not
  embedded in character cards.
